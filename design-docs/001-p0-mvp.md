我建议 **P0 不要做成"日志系统"**，而是做成一个**企业级 Audit Runtime（审计运行时）**。

P0 的目标只有一句话：

> **任何 Core 模块，都能以一行代码完成审计，并且所有审计记录都能在统一界面查看。**

注意，这里的定位不是 Elasticsearch、不是 Loki、不是普通 Log，而是**业务审计（Business Audit）**。

---

# Phase 0：Audit Runtime（MVP）

```
core-audit
```

目标：

> 建立整个 Core Platform 的统一审计基础设施。

所有 Core 模块：

```
core-user
core-config
core-storage
core-openapi
core-workflow
core-ai
core-notification
core-billing
```

统一调用：

```java
auditService.record(...)
```

而不是：

```java
log.info(...)
```

二者定位完全不同。

| log   | audit    |
| ----- | -------- |
| 调试    | 合规       |
| 程序员看  | 管理员看     |
| 可删除   | 不建议修改    |
| 文本    | 结构化数据    |
| 无业务语义 | 有业务语义    |
| Trace | Business |

# 七、Audit SDK

这是整个 Runtime 最重要的。

所有模块：

只依赖：

```
Audit SDK
```

例如：

```java
auditService.record(
    AuditEvent.builder()
        .module(Module.USER)
        .action(Action.DELETE)
        .targetType("USER")
        .targetId("1001")
        .operator(currentUser)
        .result(Result.SUCCESS)
        .description("删除用户")
        .build()
);
```

整个业务：

只有这一行。

以后：

内部实现怎么变：

业务不知道。

# 十三、MVP 不做的能力（刻意留白）

为了保持 `core-audit` 的职责单一，以下能力建议全部放到后续 Phase，而不是在 P0 提前实现：

| 能力                  | 放到 Phase | 原因              |
| ------------------- | -------- | --------------- |
| Before / After Diff | P3       | 需要统一对象快照和字段差异模型 |
| Timeline            | P5       | 依赖事件关联能力        |
| Replay              | P6       | 需要请求与操作链路的完整记录  |
| Event Bus           | P1       | 审计事件需要标准化后再发布   |
| 风险评分                | P8       | 属于智能分析，而不是基础记录  |
| AI Root Cause       | P8       | 依赖完整审计数据积累      |
| 数据保留策略              | P7       | 企业合规需求，MVP 不必引入 |
| 多租户隔离               | P7       | 与企业级权限和组织模型一起演进 |

# 十四、P0 的核心设计原则（最重要）

整个 `Audit Runtime` 在 MVP 阶段，应坚持以下原则：

1. **结构化优先，而不是文本日志。** 所有字段可检索、可统计。
2. **统一入口。** 所有 Core 模块只能通过 `Audit SDK` 写入审计记录，禁止各自实现。
3. **枚举标准化。** `Module`、`Action`、`Result` 使用统一枚举，避免数据污染。
4. **业务与审计解耦。** 业务只负责调用 `auditService.record()`，审计存储、查询、扩展全部由 `core-audit` 接管。
5. **为后续演进预留扩展点。** `metadata` 使用 JSON 保存扩展信息，详情页预留展示区域，保证未来加入 Diff、Timeline、Replay 等能力时无需推翻 P0 的模型。

这样的 P0 足够简单，但已经能作为整个 Core Platform 的统一审计基石，并能平滑演进到后续的企业级审计能力。
